# Agent-Browser Plan

## 定位

`agent-browser` 在本方案中不是唯一回归执行器，而是 Playwright 的辅助层。它的核心价值在于让工程师先用真实浏览器流程确认页面行为，再把稳定路径转成脚本化回归测试。

## 主要职责

- 探查真实页面交互路径
- 记录稳定的文本锚点与选择器候选
- 在脚本编写前做一次人工代理走查
- 在 CI 或本地回归失败后复现问题
- 补充截图、页面状态、失败步骤说明

## 推荐使用时机

编写测试前：

- 用 `agent-browser` 从 `/users` 首页走一遍核心 CRUD 流程
- 确认弹窗标题、按钮文本、表格呈现、分页文案是否稳定

脚本实现中：

- 用于确认某个断言点是否容易抖动
- 用于验证某个交互是否依赖真实浏览器行为，例如 `confirm` 或对话框焦点

回归失败后：

- 复现场景
- 记录失败截图
- 比对预期与真实页面差异

## 观察点优先级

- 文本锚点优先于脆弱 CSS 路径
- 可见按钮状态优先于内部组件结构
- 对话框标题优先于层级复杂的弹窗 DOM
- 表格单元格文本优先于图标按钮内部结构

## 任务模板

### 模板 1：列表加载走查

目标：

- 确认 `/users` 在真实环境下可加载，并识别稳定断言点

前置数据：

- 后端可用

浏览器操作序列：

1. 打开 `/users`
2. 等待页面稳定
3. 观察表头、空态、数据行、分页区域

观察点：

- `Users` 标题
- 表头文本
- `No users found` 或数据行
- `Previous` / `Next`

成功判定：

- 页面进入稳定可读状态，且存在可复用文本锚点

失败时应记录的信息：

- 页面截图
- 失败提示文本
- 当前 URL

### 模板 2：创建用户走查

目标：

- 确认创建流程、弹窗字段和成功反馈稳定

前置数据：

- 准备唯一 `e2e` 用户数据

浏览器操作序列：

1. 打开 `/users`
2. 点击 `Add User`
3. 填写 `UID`、`Name` 及可选字段
4. 点击 `Save`

观察点：

- `Create User` 标题
- `Save` 按钮状态
- 成功 toast
- 新用户是否出现在列表

成功判定：

- 创建完成且列表可见新用户

失败时应记录的信息：

- 表单字段值
- 错误提示
- 保存前后页面截图

### 模板 3：单字段编辑走查

目标：

- 确认新版 `PATCH` 语义在 UI 结果上成立

前置数据：

- 存在可编辑测试用户，记录原始值

浏览器操作序列：

1. 打开目标用户编辑弹窗
2. 仅修改 `Company`
3. 点击 `Save`
4. 刷新或重新打开弹窗确认

观察点：

- `Edit User` 标题
- 未修改时 `Save` 是否禁用
- 修改后 `Save` 是否可用
- 更新后仅目标字段变化

成功判定：

- 修改字段更新，未修改字段保持原值

失败时应记录的信息：

- 修改前后字段值
- 列表截图
- 弹窗截图

### 模板 4：清空 nullable 字段走查

目标：

- 验证清空 `email` 或 `birth` 后不会回填旧值

前置数据：

- 目标用户的 `email` 或 `birth` 初始非空

浏览器操作序列：

1. 打开 `Edit User`
2. 清空字段
3. 保存
4. 再次进入编辑弹窗确认

观察点：

- 提交前字段已清空
- 提交后列表显示为空
- 再次打开时字段仍为空

成功判定：

- 清空结果稳定持久

失败时应记录的信息：

- 清空前后截图
- 字段重新回填的具体表现

### 模板 5：删除走查

目标：

- 验证删除确认与取消删除两个分支

前置数据：

- 存在专门的临时测试用户

浏览器操作序列：

1. 点击删除按钮
2. 分别执行确认和取消两个分支

观察点：

- 浏览器确认框行为
- 删除成功 toast
- 用户行是否消失

成功判定：

- 确认后删除成功，取消后数据保留

失败时应记录的信息：

- 删除前后列表截图
- 实际触发的分支

## 与 Playwright 的协作方式

- `agent-browser` 先走查一遍真实路径，帮助沉淀稳定锚点
- Playwright 再把这些稳定路径固化成可回归脚本
- 回归失败时，再用 `agent-browser` 做人工代理复现，补充诊断信息
