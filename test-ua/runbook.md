# Runbook

## 执行顺序

推荐按以下顺序推进：

1. 用 `agent-browser` 走通真实页面
2. 确认稳定文本锚点、弹窗行为、分页表现
3. 根据 `user-management-cases.md` 实现 Playwright 用例
4. 用真实后端执行 CRUD 主链路回归
5. 回归失败时，再使用 `agent-browser` 做复现与排障
6. 发布前使用 `agent-browser` 做一次冒烟验收

## 运行前检查项

- `NEXT_PUBLIC_API_URL` 已指向真实可用后端
- 后端 `/api/v1/users` 系列接口可访问
- 当前环境允许创建、修改、删除测试用户
- 测试账号或测试环境权限满足需求
- 已准备 E2E 测试数据命名规范
- 已约定遗留测试数据清理方式

## 推荐执行批次

首批必跑：

- 列表加载
- 创建成功
- 单字段编辑
- 清空 nullable 字段
- 删除成功

第二批补齐：

- 创建校验失败
- 删除取消
- 分页
- 列表失败后重试

## 失败排障流程

1. 记录失败用例名称和测试数据标识
2. 用 `agent-browser` 重新走一遍同路径
3. 截图保存失败页面状态
4. 记录页面提示、按钮状态、弹窗是否关闭
5. 对比预期 UI 结果与真实后端结果
6. 判断问题属于脚本脆弱性、前端回归还是后端环境异常

## 冒烟验收建议

在准备发布或联调交付前，用 `agent-browser` 做一次人工代理冒烟，至少覆盖：

- 打开列表
- 创建一条测试用户
- 编辑单字段
- 清空 `email` 或 `birth`
- 删除该测试用户

## 文档维护规则

- 页面行为变化后，优先更新 `user-management-cases.md`
- 若交互锚点变化，需同步更新 `README.md` 与 `agent-browser-plan.md`
- 若接口语义变化，需同步更新 `e2e-scope.md` 与 `playwright-mapping.md`
