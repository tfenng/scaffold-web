# Test Data Strategy

## 总体原则

真实后端下的端到端测试必须避免污染普通联调数据，同时保证每条测试可重复执行、可独立清理。

## 命名规范

测试用户统一使用 E2E 前缀：

- `uid`: `e2e_<case>_<timestamp>`
- `name`: `E2E <case> <timestamp>`
- `company`: `E2E Company <case>`

示例：

- `uid=e2e_create_20260329_201500`
- `name=E2E create 20260329_201500`

## 数据分层

建议将测试数据分为三类：

- 自建自清数据
  - 适用于创建、编辑、删除、清空字段等会修改数据的用例
- 可复用只读数据
  - 适用于仅验证列表展示的用例
- 专项造数
  - 适用于分页、多状态验证等需要固定数据规模的用例

## 各类场景的数据策略

列表加载：

- 可复用现有只读数据
- 若环境可能为空，应同时接受空列表断言

创建成功：

- 必须使用唯一 `uid`
- 测试完成后删除

创建校验失败：

- 非法输入不应依赖后续清理
- 若用重复 `uid` 校验，需提前创建基准数据

编辑成功：

- 优先使用测试专属用户
- 若复用现有用户，必须记录原值并在结束后恢复

清空 nullable 字段：

- 需确保目标用户初始 `email` 或 `birth` 非空
- 验证完成后按是否复用决定恢复或删除

删除相关：

- 删除成功必须使用临时创建用户
- 取消删除可使用任意专属测试用户

分页：

- 需要至少 `21` 条可见数据，匹配当前页面默认 `page_size=20`
- 建议预置 `25` 条以上，避免边界抖动

列表失败重试：

- 不依赖特定用户数据
- 依赖稳定的环境故障注入或服务恢复方式

## nullable 字段准备

针对 `email` 与 `birth` 清空场景，至少准备以下数据状态：

- 一条 `email` 非空的测试用户
- 一条 `birth` 非空的测试用户

若两者可落在同一用户上，也可复用同一记录，但要注意不同用例间的恢复顺序。

## 清理规则

- 所有 `uid` 以 `e2e_` 开头的临时数据应纳入测试清理名单
- 优先在单条用例结束时清理，不依赖全局统一清理
- 若执行失败导致遗留数据，后续运行前应扫描并清理过期 E2E 用户

## 风险提示

- 真实后端若存在并发联调，测试数据可能被人工修改
- 若后端没有按 `uid` 做唯一约束，需补充更强的唯一标识策略
- 若环境权限不足以删除测试数据，应改为仅在专用测试环境执行
