# User Management Cases

以下用例按优先级组织，面向 `/users` 页面。

## Case 01: 列表加载成功

前置条件：

- 后端可用
- `/api/v1/users` 可正常返回

操作步骤：

1. 打开 `/users`
2. 等待页面完成首次加载

预期 UI 结果：

- 页面显示标题 `Users`
- 表头包含 `ID`、`UID`、`Name`、`Company`、`Birth`、`Actions`
- 页面最终进入以下两种稳定状态之一：
  - 显示数据行
  - 显示 `No users found`

预期后端效果：

- 前端成功请求 `GET /api/v1/users`

清理要求：

- 无

## Case 02: 创建用户成功

前置条件：

- 后端可创建用户
- 准备唯一测试数据，例如 `uid=e2e_create_<timestamp>`

操作步骤：

1. 打开 `/users`
2. 点击 `Add User`
3. 在 `Create User` 弹窗中填写：
   - `UID`
   - `Name`
   - 可选 `Email`
   - 可选 `Company`
4. 点击 `Save`

预期 UI 结果：

- 弹窗关闭
- 出现成功提示
- 列表刷新后出现新用户

预期后端效果：

- 成功调用 `POST /api/v1/users`
- 新建记录可在列表中查询到

清理要求：

- 记录新用户 ID，测试结束后删除

## Case 03: 创建表单校验失败

前置条件：

- 打开 `Create User` 弹窗

操作步骤：

1. 不填写必填字段直接提交，或输入非法邮箱
2. 点击 `Save`

预期 UI 结果：

- 对应字段显示校验错误
- 弹窗不关闭
- 不出现创建成功提示

预期后端效果：

- 若是纯前端校验失败，应不产生成功创建结果
- 若触发后端校验，前端应展示后端返回的字段错误

清理要求：

- 无

## Case 04: 单字段编辑成功

前置条件：

- 存在一条可编辑测试用户
- 记录编辑前的 `name`、`company`、`birth` 等原始值

操作步骤：

1. 打开目标用户的 `Edit User` 弹窗
2. 仅修改 `Company`
3. 点击 `Save`

预期 UI 结果：

- 保存成功后弹窗关闭
- 成功提示出现
- 列表中该用户的 `Company` 更新为新值

预期后端效果：

- 调用 `PATCH /api/v1/users/{id}`
- 重新读取后，仅 `company` 变化，其余字段保持原值

清理要求：

- 测试结束后可恢复原值，或删除该测试用户

## Case 05: 清空 nullable 字段

前置条件：

- 存在 `email` 或 `birth` 非空的测试用户

操作步骤：

1. 打开 `Edit User`
2. 清空 `Email` 或 `Birth`
3. 点击 `Save`
4. 刷新列表，必要时重新打开编辑弹窗确认

预期 UI 结果：

- 保存成功
- 对应字段展示为空值，不再回填旧值

预期后端效果：

- `PATCH /api/v1/users/{id}` 后重新读取，`email` 或 `birth` 已被清空

清理要求：

- 若该用户需复用，测试结束后恢复字段值

## Case 06: 编辑未修改时不可保存

前置条件：

- 存在任意可编辑用户

操作步骤：

1. 打开 `Edit User`
2. 不修改任何字段

预期 UI 结果：

- `Save` 按钮不可用

预期后端效果：

- 不应产生有效更新结果

清理要求：

- 无

## Case 07: 确认删除用户

前置条件：

- 已创建一条临时测试用户用于删除

操作步骤：

1. 点击目标行删除按钮
2. 在浏览器 `confirm` 中选择确认

预期 UI 结果：

- 出现删除成功提示
- 目标用户从列表消失

预期后端效果：

- 成功调用 `DELETE /api/v1/users/{id}`
- 重新读取列表后该记录不存在

清理要求：

- 无

## Case 08: 取消删除用户

前置条件：

- 存在一条可见测试用户

操作步骤：

1. 点击删除按钮
2. 在浏览器 `confirm` 中取消

预期 UI 结果：

- 用户仍保留在列表中
- 不出现删除成功提示

预期后端效果：

- 不应产生删除结果

清理要求：

- 无

## Case 09: 分页行为

前置条件：

- 后端中准备超过一页的数据，至少 `page_size + 1` 条

操作步骤：

1. 打开 `/users`
2. 确认存在分页区域
3. 点击 `Next`
4. 再点击 `Previous`

预期 UI 结果：

- 页码文案随翻页变化
- 翻页前后表格内容变化
- 第一页时 `Previous` 不可用
- 最后一页时 `Next` 不可用

预期后端效果：

- 列表请求随页码变化重新拉取

清理要求：

- 如分页数据为专门造数，测试结束后按约定清理

## Case 10: 列表失败后重试

前置条件：

- 存在可稳定复现的列表失败条件，例如测试环境临时不可用或后端故障窗口

操作步骤：

1. 打开 `/users`
2. 触发列表加载失败
3. 观察失败提示
4. 恢复服务后点击 `Retry`

预期 UI 结果：

- 页面显示错误提示和 `Retry` 按钮
- 点击 `Retry` 后，列表恢复为正常展示状态

预期后端效果：

- 初次列表请求失败
- 重试后 `GET /api/v1/users` 成功

清理要求：

- 无
